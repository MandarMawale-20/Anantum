# Packaging Guide (Windows EXE)

This project now supports a packaged desktop flow with:

- Persistent AppData user settings
- First-run GGUF model picker
- Change-model action in UI
- Backend executable packaging scaffold

## 0. Run preflight checks

```powershell
./scripts/preflight.ps1
```

This validates the required toolchain before build:

- Python
- Rust (`rustc`, `cargo`)
- Node.js and npm
- Tauri CLI (via npx)
- PyInstaller

## 1. Build backend executable

From project root:

```powershell
./scripts/build_backend.ps1
```

Expected output:

- `frontend/dist/backend/assistant-backend/assistant-backend.exe`

## 2. Build Tauri installer

```powershell
./scripts/build_frontend.ps1
```

or run all steps:

```powershell
./scripts/build_installer.ps1
```

`build_installer.ps1` runs preflight automatically before packaging.

Installer artifacts are generated by Tauri in:

- `frontend/src-tauri/target/release/bundle/`

## 3. Optional custom installer (Inno Setup)

If you need a custom install wizard, use:

- `installer/Anantum.iss`

Compile it with Inno Setup Compiler after running the build scripts.

## Runtime behavior

- On first run, if no GGUF path is configured, user is prompted to select one.
- Selected path is saved to AppData settings file:
  - `%APPDATA%/Anantum/settings.json`
- User can change model later via **Change model** button in the widget UI.

## Notes

- Keep GGUF models external to avoid giant installer size.
- The packaged backend launcher is resolved from Tauri resources in release builds.
- In debug/dev mode, Tauri still launches `bridge/stdio_bridge.py` using system Python.
